# DEVELOPMENT_RULES.md — 開発規約
最終更新: 2026-07-09 / 適用範囲: 『ヨルノハコニワ』全Phase

## 1. 開発プロセス(不変の順序)
**設計書 → 承認 → テストデータ先行 → 実装 → Node全通過 → 実機検証 → 報告 → 承認**
- コードより先に設計書。実装より先にテスト(ルール系ロジックはテストファースト必須)
- 各マイルストーンで「動くビルド」を提示し、承認を得てから次へ
- **実機(iPhone/iPad Safari・実配信経路)の結果を最終的な正とする**。シミュレータ・PC結果だけで合格判定しない

## 2. テスト方針
- **106件回帰(T1〜T7)は常設ゲート**。エンジン・判定系に触れたら必ず `node run_node.js` を実行し、全通過を確認してからデプロイする。件数は増えてよいが減らしてはならない
- 判定コアはNode/実機で**同一コード**を実行する(runner.jsのコア共通化方式を維持)
- テストと実装が食い違ったら**突合手順**: 盤面を人手で再検証→「テストが誤り」ならテスト修正、ただし**修正理由を変更履歴に必ず記録**(黙って書き換え禁止)。ルール解釈は日本将棋連盟公式ルールを基準
- **問題データの収録前機械検証は必須**: 詰み問題は成/不成・全打点・全別解をエンジンで列挙して期待値化(人手作意のみでの収録禁止。Phase 0で人手見落とし4件の実績)
- 局面データは座標指定からの機械生成+リント7項目(玉・隣接・行き所・二歩・非手番王手・inCheck宣言・駒数)
- **match問題(次の一手・定跡)の収録検証は3層必須**(p1-007〜): ①boardRefs(解説が前提とする配置の構造化宣言をSFENと照合。**誤記を直接検出できる主層**) ②semanticAsserts(正解手が出題意図=王手/取り/タダ/道が開く等を満たすかの機械検証) ③本文マス言及スキャン(文中の「3一」等が盤面・正解手と整合)。合法性チェックのみの収録は禁止

## 3. ビルド・デプロイ運用
- **ビルドIDの手動更新は禁止**(Phase 1以降)。`tools/build.py <新ID>` が全ファイルへ自動刻印し、箇所数を検証し、106件回帰を実行してから成果物を出す
- ID体系: スパイク=`s<番号>-XXX`、本実装=`p<Phase>-XXX`。**デプロイのたびに必ず繰り上げ**
- 全HTML/CSS/JSにIDを埋め込み、起動時に**全値一致を照合**(不一致=赤警告)。照合はリトライ方式(CSS読込待ち対応)
- JSON取得は `?v=ID` + `cache:no-store` + **2段フォールバック(tests/→同階層)+絶対URLログ**
- push後は5〜10分待ってから実機確認。「更新されない」時はまずビルドIDバーを見る

## 4. Git運用
- spikes用リポジトリは**凍結**(参照のみ)。本実装は専用リポジトリ
- **本実装は平坦配置(全ファイル直下)を正式採用**(p1-003〜。iPhoneアップロードの階層平坦化事故への恒久対策)。フォルダを使う場合はローカル構成でpushし、GitHub Web UI上での移動・再編は禁止
- マイルストーン(P1-a等)ごとにタグ or コミットメッセージで明示。承認済みビルドのIDをコミットに記録
- テストデータ(tests/)・生成器(cases*.py)・READMEの変更履歴はコードと同じリポジトリで管理

## 5. 実機テスト規範
- 検証は必ず**実機+GitHub Pages実配信経路**(前作・Spike期の不具合は実配信でしか出なかった)
- 各検証ページに: ビルドIDバー / 画面内ログ / **ログをコピー** を必ず実装(iPhoneにdevtoolsがないため)
- 不具合発生時: **スクリーンショット+ログ確保が先、復旧操作(リロード等)は後**
- 性能計測中は画面ロック・アプリ切替をしない(rAF停止で数値が壊れる)
- 結果報告はREADMEのテンプレート形式(機種/iOS/BuildID/項目別判定/ログ)

## 6. 命名規則
- ファイル: `snake_case.js` / データ: `snake_case.json` / ドキュメント: `UPPER_SNAKE.md` or 日本語名.md
- JS: 関数・変数=lowerCamel、定数=UPPER_SNAKE、クラス=PascalCase。エンジンはUMD・DOM非依存
- 将棋表記: 局面=SFEN、着手=USI(`7g7f`/`8h2b+`/`P*5e`)で統一。盤内部は file1-9 × rank1-9(rank1=a=一段目)
- テストID: `T<群>-<連番>`、反則は`C<類>-<連番>`。理由コードは7種固定(PROJECT.md §5)
- セーブ: 外装 `{app,schemaVersion,savedAt,crc32,body}`。スキーマ変更は必ずversion繰り上げ+マイグレーション関数

## 7. スコープ防衛
- Phase設計書の「含めないもの」リストを尊重。前倒し実装は提案→承認を経る
- 完成が危うい場合の削減順(承認済): 自室 → サブクエ数 → 裏ダンジョン → エリア統合
- 死守8項目(PROJECT.md §3)はいかなる削減でも対象外
- 抽象化はSubjectModule契約の1点のみ(YAGNI)。2教科目が現実になるまで汎用化を広げない

## 8. コンテンツ規範
- **完全オリジナル**。既存作品・キャラクター・設定・歌詞等の使用禁止
- 恐怖過多・流血・残酷描写の禁止。「不気味だけど可愛い」(敵は目がまるい・どこか間抜け)。敵は浄化して元に戻す
- 低学年対応: ふりがな/ひらがなモード、1画面の文字量最小、時間制限なし(上級の任意モード除く)
- 学習との整合: 不正解は必ず「ヒント→再挑戦→解説」で学びに変える。正しい別解は必ず正解にする

## 9. セーブ・データ保全
- 書き出し/読み込み機能は最初期から搭載(消失事故対策)。ゲーム内で週1回の書き出しを自然に促す
- 起動モード(Safari/PWA)を常時判定・表示。プライベートブラウズは警告
- セーブ目標500KB以下(実測上限5MB未満)。5000問成績は数値配列の圧縮表現
