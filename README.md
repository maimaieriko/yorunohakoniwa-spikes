# ヨルノハコニワ Phase 0 / Spike 1
9×9盤面表示・タッチ操作・GitHub Pagesキャッシュ対策の検証スパイク

> 使い捨ての検証コードです。本実装(Phase 1)はこの検証結果を反映してゼロから書きます。

---

## 1. 公開手順(新規リポジトリ)

1. GitHubで**新規リポジトリ**を作成(例: `yorunohakoniwa-spikes`)※前作とは完全に別にする
2. このフォルダの4ファイル(`index.html` / `style.css` / `app.js` / `README.md`)をリポジトリ直下にアップロード
3. リポジトリの **Settings → Pages → Branch: main / (root)** を選択して保存
4. 数分後、以下のURLで開けます:

```
https://<あなたのGitHubユーザー名>.github.io/<リポジトリ名>/
```

> **注意: GitHub Pagesは反映に数分かかることがあります。** push直後に古い表示が出ても、まず5〜10分待ってから再確認してください(それでもビルドIDが古い場合こそが、このスパイクが検出したいキャッシュ問題です)。

### 入力処理の方針(前作バグ対策)
入力処理は**Pointer Eventsに一本化**し、Safari既定動作抑止のため**touchstartはpreventDefaultのみ**に使用する(touch/clickイベントで操作処理を書かない=二重発火の根絶)。

## 2. キャッシュ対策の仕組みと運用ルール(最重要)

### 仕組み
- `index.html` が CSS/JS を **`?v=ビルドID` 付き**で読み込む → IDを変えるとSafariは別ファイルとして再取得する
- HTML / CSS / JS の**3ファイルそれぞれにビルドIDを埋め込み**、起動時に照合
- 3つが一致しなければ画面上部に**赤い警告**が出る → 「古いJSが残っている」を目視で検出できる(前作の問題の可視化)

### 更新時のルール(毎回必ず)
ファイルを1つでも変更したら、**次の4箇所のビルドIDを同じ新しい値に更新**してからpushする:

| 場所 | 行 |
|---|---|
| index.html | `window.BUILD_HTML = 's1-001';` |
| index.html | `style.css?v=s1-001` と `app.js?v=s1-001` の2箇所 |
| app.js | `const BUILD_JS = 's1-001';` |
| style.css | `#css-build-probe::after { content: 's1-001'; }` |

例: `s1-001` → `s1-002` → `s1-003` …

※本実装ではこの付与をビルドスクリプトで自動化します。スパイクでは「この方式でSafariのキャッシュを確実に破れるか」自体が検証対象のため、あえて手動です。

## 3. iPhone/iPad Safariでのテスト項目

### A. 表示
- [ ] A-1: 盤が9×9で表示され、画面からはみ出さない(縦持ち)
- [ ] A-2: 横持ちに回転しても崩れない
- [ ] A-3: 駒の文字が読める。後手の駒が180度回転している
- [ ] A-4: 筋(9〜1)・段(一〜九)のラベルが表示される
- [ ] A-5: ノッチ/ホームバーに内容が隠れない
- [ ] A-6: 「盤サイズ」ボタンでA/Bを切替え、**iPhoneとiPadそれぞれで見やすい方**を記録する(A: 52px上限/駒0.58、B: 60px上限/駒0.62)
- [ ] A-7: 起動直後のログに「**セルフテスト完了: 全12件OK**」と出る(進行方向・ブロック処理の自動検証)

### B. タッチ操作
- [ ] B-1: 先手の歩をタップ → 前のマスが青く光る → 光るマスをタップで移動できる
- [ ] B-2: 飛・角・香が途中の駒を飛び越えない(光るマスが正しい)
- [ ] B-3: 相手の駒がいるマスは赤く光り、タップで取れる
- [ ] B-4: 選択中にもう一度同じ駒をタップすると選択解除される
- [ ] B-5: 選択中に自分の別の駒をタップすると選び直しになる
- [ ] B-6: 動けないマスをタップすると一瞬赤く点滅し、何も起きない
- [ ] B-7: 端のマス(1筋・9筋)も指で確実に押せる。狙いと違うマスに反応しない
- [ ] B-8: **盤を素早く連打**しても固まらない(固まった場合「選択解除」で復帰するか)
- [ ] B-9: **2本指で同時タッチ**しても誤動作しない(ログに「多点タッチを無視」と出る)
- [ ] B-10: 手番チェックONで先手→後手→先手…と交互にしか動かせない/OFFで自由に動かせる

### C. iOS Safari既定動作の抑止
- [ ] C-1: 盤の上で**ダブルタップしても拡大しない**
- [ ] C-2: 駒を**長押ししても**選択メニュー・コピーが出ない
- [ ] C-3: 盤の上で指を動かしてもページがスクロールしない(盤の外ではスクロールできる)
- [ ] C-4: ピンチしてもレイアウトが壊れない

### D. キャッシュ対策(最重要)
- [ ] D-1: 初回表示で ビルドID HTML / JS / CSS が3つとも同じ値
- [ ] D-2: **更新テスト**: ビルドIDを `s1-002` に上げてpush → 数分後にiPhoneのSafariで**普通に再読み込み** → 3つとも `s1-002` になる(赤い警告が出ない)
- [ ] D-3: タブを閉じて開き直した場合も新IDになる
- [ ] D-4: (可能なら)ホーム画面に追加した場合の挙動も確認

### 報告のお願い
- 問題があった項目の番号+スクリーンショット
- 「ログをコピー」ボタンでログを取得して貼り付け(端末情報と操作履歴が入ります)

## 4. このスパイクの範囲外(仕様であり不具合ではない)

- 成り/不成り、持ち駒(駒台)、打ち
- 王手放置の禁止・詰み判定(→ V1/V2スパイクで検証)
- 二歩などの反則判定(→ V1)
- 勝敗判定・対局進行(王を取ってもゲームは続きます)
- 音、アニメーション、セーブ
- `prefers-reduced-motion` 対応(→ Phase 6で対応予定)
- ピンチ拡大の無効化(`user-scalable=no`)はスパイク限定の措置。製品版ではアクセシビリティの観点から再検討する
